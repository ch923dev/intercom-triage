## CRITICAL

### C1 — AI `resolved` verdict writes illegal `resolved_source` → entire ingest batch crashes (500)
**`app/services/tickets.py:174`** (`_maybe_auto_resolve_from_ai`), constraint **`app/models.py:522-526`**.

`_maybe_auto_resolve_from_ai` accepts verdict in `("resolved", "non_actionable")` (line 160) then does:
```python
row.resolved_source = result.ai_resolution_verdict   # line 174
```
The CheckConstraint `tickets_resolved_source_check` allows only `('manual','intercom_closed','non_actionable')`. When the AI returns `resolution_verdict='resolved'`, this writes `resolved_source='resolved'` → `session.commit()` (line 323) raises `IntegrityError: CHECK constraint failed`. This aborts the **whole batch** — every ticket in that sync fails, extension sync returns 500.

This is the single most common auto-resolve path ("customer says thanks, fixed"). Confirmed by agent repro **and** by main-thread code read.

**Why it slipped through:** mypy can't catch it (ORM column is `Mapped[str | None]`, not a `Literal`). The only ingest auto-resolve test uses verdict `'non_actionable'` — which coincidentally doubles as a valid `resolved_source`. The `'resolved'` verdict is never routed through this helper in any test.

**Fix (contract decision — touches schema + model + webapp, so deliberate):** map verdict → source explicitly instead of assigning verdict as source. Either add an `'ai_resolved'` value to `ResolvedSource` (`schemas.py:56`) + the CheckConstraint + webapp consumers, or map `'resolved'` → an existing source. Then add the missing test.

## HIGH

### C2 — AI auto-resolve silently undoes a manual reopen on the next sync
**`app/services/tickets.py:158, 177-246`.**

`_maybe_auto_resolve_from_ai` runs on **every** re-sync of an open ticket; its only re-resolve guard is `if row.resolved_at is not None: return`. After an operator drags a resolved ticket out of Resolved (`set_override` clears `resolved_at`) or hits `/reopen`, the ticket is open with `resolved_at=None`. Next sync, the **cached** high-confidence verdict is read back and re-stamps `resolved_at` — the ticket bounces back to Resolved, overriding the operator. Invariant #11 (atomic reopen) holds for the immediate transaction but is **not durable** against the next ingest. No sticky-flag equivalent to `title_user_edited` exists for resolution state. Untested.

**Fix:** suppress auto-resolution when the operator has acted — e.g. a sticky `resolution_user_cleared` flag set by reopen/drag-out, cleared only when the customer-visible thread genuinely advances (new part timestamp > clear time).

## MEDIUM

### M1 — `POST /tickets/ingest` accepts an unbounded `list[HydratedTicket]`
**`app/routers/tickets.py:61`**, service **`tickets.py:248-325`.** No server-side cap on batch size, no per-part body-size limit. `MAX_BULK_IDS=200` guards bulk endpoints but **not ingest**. Fully buffered, then every uncached ticket fans out to OpenRouter — memory + token-cost amplifier. Localhost-only, so robustness/cost issue, not remote DoS.
**Fix:** `Field(max_length=...)` on the ingest list; consider a `max_length` on `ConversationPartSchema.body`.

### M2 — Blocking file / PIL I/O inside async handlers
**`app/services/attachments.py:62`** (`write_bytes`), **`:115-135`** (PIL `open/convert/thumbnail/save`). Run synchronously on the event loop in `async def` service fns — stalls concurrent requests on large images. No `anyio.to_thread` anywhere in `app/`. The repo selects ruff's `ASYNC` ruleset specifically to catch this; plain pathlib/PIL slips past it. Low practical impact for one operator.
**Fix:** wrap write + PIL pipeline in `await anyio.to_thread.run_sync(...)`.

### M3 — Single-id `set_override` accepts an unknown ticket; bulk path rejects it
**`tickets.py:set_override`** 404s only on bad *category* — `session.get(Ticket,…)` may be `None` and an `Override` row is still created (no FK to tickets). `bulk_recategorize` (`bulk.py:141`) 404s per-id on missing ticket. Inconsistent contract; single-id silently creates an orphan override. Not data-corrupting.

## LOW / nits

- **L1 — Stale cache on timestamp regression / empty parts.** `_content_signature` (`tickets.py:81`) = `max(part.created_at)`, no content hash. A same-timestamp edited body, a regressed timestamp, or an empty-`parts` ticket can serve a stale category until TTL. Edge case; Intercom timestamps are normally immutable.
- **L2 — Misleading test mock.** `tests/test_resolution_ingest.py:56` sets `openrouter.classify = AsyncMock(...)` but the real method is `complete`. The AI mock is dead; the test actually exercises the fallback path, not the verdict path it names. (Same root cause as the missing C1 coverage.) Produces the 2 benign `coroutine never awaited` warnings.
- **L3 — Title truncation mismatch.** `edit_ticket` truncates title `[:120]` (`tickets.py:350`) but schema allows `max_length=200` (`schemas.py:383`). 121–200-char titles silently truncated.
- **L4 — Two config-access paths.** `routers/health.py:18` uses `get_config` (lru-cached) while every other router uses `get_app_config` (`app.state`). Same value today; could diverge.
- **L5 — `Retry-After` HTTP-date form ignored** (`clients/openrouter.py:47`) — only numeric seconds parsed; OpenRouter sends numeric, so fine.
- **L6 — Dead field** `_ResolverState.fallback_category_id` (`pipeline.py:196`) set but never read.
